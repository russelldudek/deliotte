The actual mandate

Reduce the distance between AI ambition and a production-capable workflow.

Deloitte is not hiring a presentation layer around frontier models. The FDE must identify valuable work, build rapidly, communicate with senior stakeholders, make engineering tradeoffs visible, and leave behind reusable assets plus client-operable capability.

Move at frontier-model speed while delivering at enterprise-control standards.
01

Speed vs governance

Prototype quickly without deferring permissions, failure behavior, evaluation, economics, or ownership until the end.

02

Client specificity vs reusable delivery IP

Solve the actual workflow while capturing patterns, code, evaluations, prompts, and runbooks that improve the next engagement.

03

Model capability vs system reliability

A strong single response is not a reliable operating system. Repeated behavior, tool state, latency, and recovery matter.

04

Agent autonomy vs human authority

Autonomy must be earned by consequence, evidence, permissions, and reversibility—not assumed from model fluency.

Forward Deployment Contract

Advance the workflow only when its next obligation is satisfied.

The operating model is a deployment contract, not a generic innovation funnel.

01

Frame

Define the changed work, affected user, current failure mode, value hypothesis, baseline, constraints, and accountable owner.

Exit: a workflow problem, not an AI feature request.
02

Prove

Build the smallest end-to-end implementation that tests context, tool use, model behavior, workflow fit, and user usefulness.

Exit: a real user completes meaningful work.
03

Bound

Define permissions, approvals, escalation, refusal, evaluation, versioning, cost, latency, and model-risk limits.

Exit: allowed, prohibited, and held actions are explicit.
04

Embed

Integrate identity, APIs, workflow systems, logging, observability, CI/CD, support, and operating cadence.

Exit: capability works inside the real workflow.
05

Transfer

Complete runbook, ownership, training, architecture record, reusable assets, backlog, and performance review cadence.

Exit: the client can operate and improve it.

First 90 days

Earn technical trust, own a bounded workstream, then prove transferability.

Days 1–30

Learn the pod and ship

  • Learn approved platforms, architecture, security, evaluation, code, and documentation standards.
  • Pair on client framing and working sessions.
  • Build small tool-use and retrieval implementations.
  • Contribute tests, documentation, runbooks, and issue resolution.
  • Target: first merged engineering contribution.
Days 31–60

Own a bounded workstream

  • Define the workflow baseline and value hypothesis.
  • Implement one end-to-end AI-enabled workflow with a realistic integration boundary.
  • Define permissions, approvals, failure tests, and known limitations.
  • Instrument latency, cost, model behavior, and tool outcomes.
  • Target: representative users complete meaningful work.
Days 61–90

Prove operability and transfer

  • Support production-readiness and CI/CD decisions.
  • Create runbook, monitoring, escalation, ownership, and backlog.
  • Pair with client engineers and operators.
  • Convert repeated elements into reusable assets.
  • Target: client-operable workflow and reusable Deloitte capability.
The hard objection
“Is someone with broad operations and leadership scope genuinely prepared to code, test, debug integrations, document, and contribute inside a pod?”
What the concern protects

The pod needs a practitioner who contributes directly, accepts technical feedback, and does not substitute seniority for working software.

My answer

My titles describe the scope of problems for which I have been accountable—not a preference to remain distant from the work. I am pursuing this role to be closer to the build and the client problem.

The proof required

Judge the work: workflow logic, technical reasoning, evaluation discipline, explicit authority, operational controls, and the willingness to state where production-platform evidence is not yet verified.
