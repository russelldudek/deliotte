Entry principle

Earn technical trust first. Own a bounded workstream next. Prove client operability and reusable delivery value by day 90.

Days 1–30 — Learn the pod and ship

Client and workflow

  • Join senior FDEs in use-case discovery and working sessions.
  • Map the changed work, user, baseline, constraints, decision rights and client owner.
  • Listen for the difference between the nominal request and the actual workflow problem.

Engineering system

  • Review approved frontier platforms, reference architectures, security and evaluation standards.
  • Reproduce tool-use, retrieval, observability and deployment patterns.
  • Learn code-review, testing, documentation and release expectations.

Direct contribution

  • Build small tool-use and retrieval implementations.
  • Add tests, documentation, runbook improvements and issue resolution.
  • Target the first merged engineering contribution and specific mentor feedback.

30-day evidence

Pod trust

Reviewed contribution merged; feedback incorporated without defensiveness.

Client understanding

Clear workflow, stakeholder, authority and constraint map.

Platform fluency

Working implementation using the pod’s preferred platform and patterns.

Operating fit

Reliable participation in sprint sizing, documentation and issue follow-through.

Operating posture

Make uncertainty visible. Treat evaluation and documentation as part of the build, not cleanup. Ask where human authority sits before increasing agent autonomy. Seek small, reviewable technical commitments rather than performing seniority.

Days 31–60 — Own a bounded workstream

  • Define one workflow baseline, value hypothesis, representative user and failure mode.
  • Implement one end-to-end AI-enabled workflow with a realistic API or enterprise-system boundary.
  • Define permissions, human approval, refusal, escalation and rollback behavior.
  • Build representative, adversarial, repeated-run and tool-failure evaluation cases.
  • Instrument latency, cost, model behavior, tool outcomes, retries and overrides.
  • Document architecture decisions, known limitations and the production-hardening backlog.
60-day evidence

Representative users complete meaningful work; the decision path, authority model, evaluation results, limitations and observable behavior are documented.

Days 61–90 — Prove operability and transfer

  • Support deployment-readiness and CI/CD decisions.
  • Create the runbook, monitoring plan, escalation logic, ownership map and backlog.
  • Pair with client engineers and operators until routine operation and diagnosis are independent.
  • Convert repeated implementation elements into reusable modules, evaluations, prompts or reference patterns.
  • Present a concise outcome review connecting technical evidence to the business workflow and client operating ownership.

Day-90 standard

Business outcome

Baseline and review cadence exist for the changed workflow.

Quality and reliability

Representative evaluation and repeated-run behavior are visible.

Human authority

Approval, escalation and consequential-action boundaries are explicit.

Operability

Logs, cost, latency, recovery and support behavior are diagnosable.

Client transfer

A named team can operate, evaluate and improve the workflow.

Deloitte reuse

The engagement leaves a reusable asset, evaluation or delivery lesson.
